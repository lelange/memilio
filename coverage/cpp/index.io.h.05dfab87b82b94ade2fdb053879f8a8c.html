20 /**
21 * This file describes the framework for IO. The framework is format-independent,
22 * all the format specific implementations are in the epidemiology_io library.
23 *
24 * Main functions and types:
25 * -------------------------
26 * - functions serialize and deserialize:
27 * Main entry points to the framework to write and read values, respectively. The functions expect an IOContext
28 * (see Concepts below) that stores the serialized data. (De-)serialization can be customized by providing a
29 * (de-)serialize_internal overload or a (de-)serialize member function for the type. See the section "Implementing
30 * serialization for a new type" or the documentation for `serialize` and `deserialize`.
31 * - IOStatus and IOResult:
32 * Used for error handling, see section "Error Handling" below.
33 *
34 * Concepts:
35 * ---------
36 * 1. IOContext
37 * Stores data that describes serialized objects of any type in some unspecified format and provides structured
38 * access to the data for deserialization. Implementations of this concept may store the data in any format
39 * they want including binary. The data may also be written directly to disk. The context also keeps track
40 * of errors. An IOContext object `io` allows the following operations:
41 * - `io.create_object("Type")`:
42 * Returns an IOObject for the type called `"Type"`. The IOObject (see below) allows adding data that describes
43 * the object to be serialized. The function must return something that can be assigned to a local
44 * variable, e.g., a temporary or copyable function. IOObject may store references to the context internally,
45 * so the lifetime of the local IOObject may not exceed the lifetime of the IOContext that created it.
46 * - `io.expect_object("Type")`:
47 * Returns an IOObject for the type called `"Type"`.
48 * The IOObject (see below) provides access to the data needed for deserialization.
49 * - `io.flags()`:
50 * Returns the flags that determine the behavior of serialization; see IOFlags.
51 * - `io.error()`:
52 * Returns an IOStatus object to check if there were any errors during serialization.
53 * Usually it is not necessary to check this manually but can be used to report the error faster and
54 * avoid expensive operations that would be wasted anyway.
55 * - `io.set_error(s)` with some IOStatus object:
56 * Stores an error that was generated outside of the IOContext, e.g., if a value that was deserialized
57 * is outside an allowed range.
58 * 2. IOObject
59 * Gives structured access to serialized data. During serialization, data can be added with `add_...` operations.
60 * During deserialization, data can be retrieved with `expect_...` operations. Data must be retrieved in the same order
61 * as it was added since, e.g., binary format does not allow lookup by key. The following operations are supported
62 * for an IOObject `obj`:
63 * - `obj.add_element("Name", t)`:
64 * Stores an object `t` in the IOObject under the key "Name". If `t` is of basic type (i.e., int, string), IOObject
65 * is expected to handle it directly. Otherwise, the object uses `epi::serialize` to get the data for `t`.
66 * - `obj.add_list("Name", b, e)`:
67 * Stores the elements in the range represented by iterators `b` and `e` under the key "Name". The individual elements are not named.
68 * The elements are either handled directly by the IOObject or using `epi::serialize` just like `add_element`.
69 * - `obj.add_optional("Name", p)`:
70 * Stores the element pointed to by pointer `p` under the key "Name". The pointer may be null. Otherwise identical to add_element.
71 * - `obj.expect_element("Name", Tag<T>{})`:
72 * If an object of type T can be found under the key "Name" and can be deserialized, returns the object. Otherwise returns
73 * an error. Analogously to serialization, the IOObject is expected to handle basic types directly and use `epi::deserialize`
74 * otherwise.
75 * - `obj.expect_list("Name", Tag<T>{})`:
76 * If a list of objects of type T can be found under the key "Name" and can be deserialized, returns a range that can be
77 * iterated over. Otherwise returns an error.
78 * - `obj.expect_optional("Name", Tag<T>{})`:
79 * Returns boost::optional<T> if an optional value of type T can be found under the key "Name". The optional may contain a
80 * value or it may be empty. Otherwise returns an error. Note that for some formats a wrong key is indistinguishable from
81 * an empty optional, so make sure to provide the correct key.
82 *
83 * Error handling:
84 * ---------------
85 * Errors are handled by returning error codes. The type IOStatus contains an error code and an optional string with additional
86 * information. The type IOResult contains either a value or an IOStatus that describes an error. Operations that can fail return
87 * an IOResult<T> where T is the type of the value that is produced by the operation if it is succesful. Except where necessary
88 * because of dependencies, the framework does not throw nor catch any exceptions. IOContext and IOObject implementations are
89 * expected to store errors. During serialization, `add_...` operations fail without returning errors, but the error is stored
90 * in the IOObject and subsequent calls are usually no-ops. During deserialization, the values produced must usually be used or
91 * inspected, so `expect_...` operations return an IOResult. The `apply` utility function provides a simple way to inspect the result
92 * of multiple `expect_...` operations and use the values if all are succesful. See the documentation of `IOStatus`, `IOResult`
93 * and `apply` below for more details.
94 *
95 * Implementing serialization for a new type:
96 * ------------------------------------------
97 * Serialization of a new type T can be customized by providing _either_ member functions `serialize` and `deserialize` _or_ free functions
98 * `serialize_internal` and `deserialize_internal`.
99 *
100 * The `void serialize(IOContext& io)` member function takes an IO context and uses `create_object` and `add_...` operations
101 * to add data. The static `IOResult<T> deserialize(IOContext& io)` member function takes an IO context and uses `expect_...`
102 * operations to retrieve the data. The `apply` utility function can be used to inspect the result of the `expect_...`
103 * operations and construct the object of type T.
104 * E.g.:
105 * ```
106 * struct Foo {
107 * int i;
108 * template<class IOContext>
109 * void serialize(IOContext& io) {
110 * auto obj = io.create_object("Foo");
111 * obj.add_element("i", i);
112 * }
113 * template<class IOContext>
114 * static IOResult<Foo> deserialize(IOContext& io) {
115 * auto obj = io.expect_object("Foo");
116 * auto i_result = obj.expect_element("i", epi::Tag<int>{});
117 * return epi::apply(io, [](auto&& i) { return Foo{i}; }, i_result);
118 * }
119 * };
120 * ```
121 *
122 * The free functions `serialize_internal` and `deserialize_internal` must be found with argument dependent lookup (ADL).
123 * They can be used if no member function should or can be added to the type. See the code below for examples where
124 * this was done for, e.g., Eigen3 matrices and STL containers.
125 *
126 * Adding a new serialization format:
127 * ----------------------------------
128 * Implement concepts IOContext and IOObject that provide the operations listed above. Your implemenation should handle
129 * all built in types as well as std::string. It may handle other types (e.g., STL containers) as well if it can do so
130 * more efficiently than the provided general free functions. See the JsonSerializer implementation in the epidemiology_io
131 * library for an example.
132 */
